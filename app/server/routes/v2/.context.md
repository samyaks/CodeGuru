## purpose

The v2 HTTP surface mounted under `/api/v2/projects/:id/...`:

- `gaps.js` — `GET /gaps`, `POST /gaps/:gapId/{accept,reject,restore,refine,mark-committed}`
- `shipped.js` — `GET /shipped`, `POST /shipped/:itemId/reopen`

Mounted from `app/server/app.js`. v1 endpoints (`/api/suggestions`,
`/api/projects/:id/...`) stay alive in parallel until Phase 6b cutover.

## constraints

- Use `asyncHandler` from `app/server/lib/async-handler.js` to wrap
  every route. Don't catch errors manually — let `AppError` thrown from
  `lib/app-error.js` flow to the global error handler.
- Authorize every project-scoped route with
  `lib/helpers.js#checkProjectAccess`. Never trust `req.params.id`
  without it.
- Use `lib/rate-limit.js#createRateLimit` for write paths. Refine has
  its own tighter limit because each call costs an Anthropic request.
- Mount with `optionalAuth(supabase)` (so public, `user_id IS NULL`
  projects work without login). The `checkProjectAccess` helper is
  what enforces ownership when `user_id` is set.
  *(Footgun: this currently lets unauthenticated users mutate state
  on public projects. See followup #7.)*
- Translate kebab-case status values from the URL/body to snake_case
  before hitting the DB, and vice versa on the way out. The mapper
  helper lives in `app/server/services/v2/gap-mapper.js`.
  *(`?status=` query param does NOT yet do this — see followup #6.)*

## decisions

### One router per resource, mounted with `mergeParams: true`
Both `gaps.js` and `shipped.js` use `Router({ mergeParams: true })` so
they can read `req.params.id` (the project id) from the parent mount
path. Same pattern as `routes/projects.js` sub-routers.
agent · 2026-05-04

### `pruneInternalFields` strips internal-only gap fields
The v2 Gap shape produced by `gap-mapper.js#toGap` carries
`rawCategory`, `priority`, `type`, `affectedFiles` for matcher use.
Routes prune those before responding so the API surface stays minimal
and refactoring internal fields doesn't break clients.
agent · 2026-05-04

### Reopen creates a brand-new untriaged gap, not a status flip
When a `partial`-verification ShippedItem is reopened, we insert a new
row scoped to the still-failing files rather than reverting the
original suggestion's `v2_status`. This preserves the audit trail of
what was attempted and keeps the Shipped feed historically accurate.

because `suggestions.createBatch` re-hashes the id. See followup #5.)*
