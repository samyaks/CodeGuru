# .context.md — v2 backend services

## owner

@samyak (PM) · agent (initial build)

## purpose

The pure logic layer behind the v2 (Takeoff) Gaps and Shipped tabs.
Lives in these files plus this doc:

- `gap-mapper.js` — translate v1 `suggestions` rows into v2 Gap shape
  consumed by the API + UI. Owns kebab/snake status translation,
  `affectedJobs` enrichment, and synthesis of map-derived gaps.
- `gap-prompt-generator.js` — wrap Claude calls that produce a Cursor
  prompt for an accepted gap (or a refined one).
- `gap-matcher.js` — match an incoming commit to an open Gap using a
  4-tier strategy (commit-ref, file-overlap, keyword, [Claude]).
- `gap-verifier.js` — given a matched (gap, commit) pair, decide
  whether the commit fully / partially / can-not-tell resolves the
  gap. Fail-safe contract: any failure → `'pending'`.
- `shipped-runner.js` — glue between webhook payloads, the matcher,
  and the verifier. Writes `shipped_items` rows + updates the source
  suggestion's `v2_status` / `verification`.
- `gap-job-linker.js` — connect persisted gaps to the product map's
  jobs-to-be-done, so the Gaps tab can show "this blocks Job X for
  Persona Y" and filter by persona.

## constraints

- All Anthropic calls go through `app/server/lib/anthropic-tracked.js`.
  Never construct a raw `Anthropic` client.
- The verifier MUST return `'pending'` (not `'partial'`) when:
  - Claude is unreachable
  - Claude returns malformed JSON
  - Claude returns a string we don't recognize as `'verified'` /
    `'partial'`
  Anything else corrupts the user's understanding of "shipped."
  *(Honored as of 2026-05-04 — see `gap-verifier.js` allowlist + followup #2.)*
- The matcher MUST NOT auto-create a `shipped_items` row when match
  confidence is below `MIN_CONFIDENCE` (0.7). *(Honored as of 2026-05-04
  — `shipped-runner.js` logs and skips. See followup #1.)*
- `shipped_items` MUST be unique per `(project_id, commit_sha)`. Enforced
  by migration 012 (UNIQUE INDEX) and `shippedItems.create`'s
  `ON CONFLICT (project_id, commit_sha) DO NOTHING`. Any new helper that
  inserts into `shipped_items` must rely on the index for correctness, not
  a check-then-insert.
- Mutating endpoints in `routes/v2/gaps.js` and `routes/v2/shipped.js`
  require an authenticated user (`requireUser` middleware). GET routes
  remain on `optionalAuth` so dashboards work without a session.
- The status enum stored in the DB is snake_case (`'in_progress'`).
  Outgoing API + UI uses kebab-case (`'in-progress'`). Translation
  lives in `gap-mapper.js#v2StatusFor`. Any new helper that reads or
  writes `v2_status` MUST go through the mapper or be tested for both
  casings.
- Don't make these files require a network or DB at import time.
  They're imported by tests and by routes that mock dependencies.

## decisions

### 4-tier matcher with explicit confidences
Rather than a single ML classifier, we use commit-ref (1.0),
file-overlap >50% (0.9), keyword + file-touch (0.8), and (planned)
Claude (≤0.7). Cheap tiers run first; we only burn an Anthropic call
when nothing else fires. The hard `MIN_CONFIDENCE = 0.7` lives in
`gap-matcher.js` so callers can't accidentally lower it.
agent · 2026-05-04

### Rule-based verifier pass before Claude
For `category === 'broken'` gaps with explicit `affectedFiles`, the
verifier first checks set-overlap. 100% overlap → `'verified'`,
partial → `'partial'` with the missing-files list. Only otherwise
does it call Claude. Saves cost on the most common gap type.
agent · 2026-05-04 · see `gap-verifier.js#ruleBasedBrokenCheck`

### `shipped-runner` runs sequentially per push
A 50-commit force-push shouldn't fan out 50 parallel Anthropic calls.
`processPush` awaits each commit before the next so we don't stampede
the verifier. Tradeoff is wall-clock latency for large pushes; we
accept that for cost predictability.
agent · 2026-05-04

### Gap → job links live on `suggestions.v2_job_links` (JSONB), not a join table
Migration 013 adds a JSONB column rather than `suggestion_jobs`. Two
reasons: the link payload depends on the *current* product map, so a FK
to `map_jobs` would break when the map is regenerated and old
suggestion rows reference deleted jobs; and the typical row has 1-3
links, so a join table 3xs the row count for the same data. JSONB is
also consistent with how the rest of v2 stores graph data (`scores`,
`entities`, `edges` are all JSONB on `product_maps`).
`gap-job-linker.js#linkGapsToJobs` writes `[]` (not `null`) when it
tries and finds nothing — `null` is reserved for "haven't tried yet"
so the GET handler can lazy-link without re-running on every reload.
agent · 2026-05-04

### Synthetic map gaps are computed at request time, not persisted
`gap-mapper.js#synthesizeMapGaps` runs on every `GET /gaps` to surface
"Build [Entity] for [Job]" rows for any `needs` edge whose target is
missing or partial. They're not in the `suggestions` table — the id
encodes `entity:job` and the lazy `/gaps/:id/prompt` endpoint regenerates
the prompt on demand. When the entity is built (status flips to
`detected`/`confirmed`/`full`), the synthetic gap auto-disappears on the
next reload. Dedupe rule: any `(jobId, *)` pair already covered by an AI
gap suppresses the synthetic gap so we don't double-surface the same
work. AI gaps are more specific (effort, severity, evidence) so we'd
rather show those.
agent · 2026-05-04 · see `gap-mapper.js#synthesizeMapGaps`

### Heuristic linker pass before Claude
`gap-job-linker.js` tries category→capability and `affected_files`→
entity matching first. A gap that classifies as `auth` is linked to any
job that needs `cap:auth` at confidence 0.85. Only if the heuristic
finds no link does the linker fall back to a single batched Claude
call. Saves cost on the most common gap categories (auth/database/etc.)
where the link is mechanical.
agent · 2026-05-04
