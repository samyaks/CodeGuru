# .context.md — v2 project pages

## owner

@samyak (PM) · agent (initial build)

## purpose

The v2 project workspace at `/v2/projects/:id`, plus the
section-component children (`GapsSection`, `ShippedSection`,
`MapSection`, `ContextSection`) and the throwaway `/v2/style-guide` QA
route.

These pages are wrapped by `pages/ProjectRouter.tsx` which selects
between v1 (`ProjectView`) and v2 (`Project`) per request based on the
`USE_V2_PROJECT` flag plus `?v1=true` / `?v2=true` overrides.

## constraints

- Tab state lives in the URL hash (`#gaps`, `#map`, `#context`,
  `#shipped`). Don't move it into a query param — `?v1=true` already
  occupies that surface for the flag escape hatch.
- API helpers go through `services/v2Api.ts` (gaps + shipped) or
  `services/productMapApi.ts` (map). Don't hand-roll fetch inside
  section components.
- `Project.tsx` must redirect to `/takeoff/:id` when project status is
  `analyzing` / `pending` (parity with v1 `ProjectView`).
- Stub out — don't fake — Claude responses. The `ChatDrawer` in
  `Project.tsx` uses canned replies; replacing them with a real stream
  is a separate phase.
- Don't add new top-level v2 pages without updating `App.tsx` AND
  `pages/ProjectRouter.tsx` if they should respect the flag.

## decisions

### Tabs by hash, not by route
`<Route path="/v2/projects/:id">` renders one component; tab switching
just rewrites `window.location.hash`. Lets v1 dashboard cards link
straight at `/projects/:id#map` without needing v2 to know about it.
agent · 2026-05-04 · see `Project.tsx#readHashTab`

### Section components own their data fetching
`GapsSection`, `ShippedSection`, `MapSection`, `ContextSection` each
load their own data from `services/v2Api.ts` and `services/api.ts`.
`Project.tsx` owns only the chrome (header, sidebar, ChatDrawer,
loading shell). This keeps the project page small and lets us
mount any section in isolation (e.g. inside the StyleGuide).
agent · 2026-05-04

### `MapSection` has its own inline editor (no wizard)
The v1 ProductMap onboarding wizard was killed in Phase 5 and replaced
with an inline editor inside `MapSection.tsx` (auto-extract CTA when no
map exists, then a per-persona expandable card for adding/removing
jobs). Don't restore links to `/projects/:id/map/onboard` — that route
now redirects back here and would create a navigation loop.
agent · 2026-05-04

### Per-job readiness uses `scores.job` + `needs` edges, not its own scoring
`JobRow` displays a 0..100% readiness chip plus a "X of Y built · N not
built yet" summary. The percent comes from `scores.job[id]` (already
rounded by `services/job-scorer.js#scoreJob`); the build/partial/missing
breakdown comes from walking `edges` of type `needs` and classifying the
target entity's `status` with the same mapping the backend uses
(`detected|confirmed|full → built`, `partial|stub → partial`, else
`missing`). When the backend mapping changes, update
`classifyEntityStatus` in `MapSection.tsx` in lockstep — the UI is the
mirror, not the source of truth.

A `needs` edge whose target entity isn't in `data.entities` is rendered
as a synthetic "Unknown component / not built" need, mirroring how
`getEntityStatus` returns 0 for an unresolved entity. This keeps the
counts honest with the percent (without it, "1/1 built · 100%" could
appear next to a 50% score).

All score reads (job, persona, app) go through `clampScore`, which
coerces strings, rejects `NaN`/`Infinity`, and clamps to `[0, 100]`. Use
it for any new readiness display so dirty data can't render as `NaN%`.
agent · 2026-05-04

### Job rows auto-expand when there's work to do
A job with any missing or partial component opens by default so the
user sees "what's not built yet" without an extra click; fully-built
jobs collapse to keep the persona card scannable. The implementation
uses a `userOverride: boolean | null` state — when `null`, expansion
re-derives from the latest counts on every render, so a refetch (e.g.
after a priority edit) correctly flips a now-built job to collapsed.
A user click sets `userOverride` so their choice sticks across
refetches without us fighting them.
agent · 2026-05-04

### `Project.tsx` keeps a `PLACEHOLDERS` map even though every tab is real
Cheap defense for when a future tab id is added to the union before
its section is wired up. The reviewer flagged this as dead code; we
keep it deliberately as the empty-state fallback.
agent · 2026-05-04

### Gaps tab is filtered by persona and shows job context
`GapsSection` now reads `data.personas` from the API and renders a
second row of filter chips below the status chips (Active / All /
Rejected). Selecting a persona narrows the visible list to gaps whose
`affectedJobs` include that persona's id. Each `GapCard` shows the
persona+job pairs as small chips ("👤 Tom · Sign up") instead of the
old free-form `affects` strings — the data comes from
`suggestions.v2_job_links` server-side via `attachAffectedJobs`. The
backend always sets `affectedJobs` (empty array when nothing's linked
yet), so the card falls back to the legacy `affects` / `required_for`
line whenever `affectedJobs.length === 0`. The GET handler kicks off a
background linker pass for any unlinked rows so the next reload has
real chips.
agent · 2026-05-04 · see `services/v2/gap-job-linker.js`

### Synthetic map gaps are read-only with lazy prompts
A "From Map" badge appears on synthetic gaps (id starts with `map-`),
generated server-side from the product map's missing/partial entities.
They have a dashed border and a single CTA — "Get Cursor prompt" —
that fetches a prompt only when the user clicks. We don't generate
prompts upfront because a project can have dozens of synthetic gaps
and most won't be acted on; running Claude eagerly would burn tokens.
Once fetched, the prompt is held in the in-memory gap row only (not
cached) — synthetic gaps disappear when the underlying entity is built,
so caching the prompt would be pointless. AI gaps keep their
Accept/Reject/Refine/Mark-committed flow because they have a real
suggestion row to mutate.
agent · 2026-05-04 · see `components/v2/GapCard.tsx#isSynthetic`
