## purpose

The v2 project workspace at `/v2/projects/:id`, plus the
section-component children (`GapsSection`, `ShippedSection`,
`MapSection`, `ContextSection`) and the throwaway `/v2/style-guide` QA
route.

These pages are wrapped by `pages/ProjectRouter.tsx` which selects
between v1 (`ProjectView`) and v2 (`Project`) per request based on the
`USE_V2_PROJECT` flag plus `?v1=true` / `?v2=true` overrides.

## constraints

- Tab state lives in the URL hash (`#gaps`, `#map`, `#context`,
  `#shipped`). Don't move it into a query param тАФ `?v1=true` already
  occupies that surface for the flag escape hatch.
- API helpers go through `services/v2Api.ts`. Don't hand-roll fetch
  inside section components.
- `Project.tsx` must redirect to `/takeoff/:id` when project status is
  `analyzing` / `pending` (parity with v1 `ProjectView`).
- Stub out тАФ don't fake тАФ Claude responses. The `ChatDrawer` in
  `Project.tsx` uses canned replies; replacing them with a real stream
  is a separate phase.
- Don't add new top-level v2 pages without updating `App.tsx` AND
  `pages/ProjectRouter.tsx` if they should respect the flag.

## decisions

### Tabs by hash, not by route
`<Route path="/v2/projects/:id">` renders one component; tab switching
just rewrites `window.location.hash`. Lets v1 dashboard cards link
straight at `/projects/:id#map` without needing v2 to know about it.
agent ┬╖ 2026-05-04 ┬╖ see `Project.tsx#readHashTab`

### Section components own their data fetching
`GapsSection`, `ShippedSection`, `MapSection`, `ContextSection` each
load their own data from `services/v2Api.ts` and `services/api.ts`.
`Project.tsx` owns only the chrome (header, sidebar, ChatDrawer,
loading shell). This keeps the project page small and lets us
mount any section in isolation (e.g. inside the StyleGuide).
agent ┬╖ 2026-05-04

### `MapSection` has its own inline editor (no wizard)
The v1 ProductMap onboarding wizard was killed in Phase 5 and replaced
with an inline editor inside `MapSection.tsx` (auto-extract CTA when no
map exists, then a per-persona expandable card for adding/removing
jobs). Don't restore links to `/projects/:id/map/onboard` тАФ that route
now redirects back here and would create a navigation loop.
agent ┬╖ 2026-05-04

### Per-job readiness uses `scores.job` + `needs` edges, not its own scoring
`JobRow` displays a 0..100% readiness chip plus a "X of Y built ┬╖ N not
built yet" summary. The percent comes from `scores.job[id]` (already
rounded by `services/job-scorer.js#scoreJob`); the build/partial/missing
breakdown comes from walking `edges` of type `needs` and classifying the
target entity's `status` with the same mapping the backend uses
(`detected|confirmed|full тЖТ built`, `partial|stub тЖТ partial`, else
`missing`). When the backend mapping changes, update
`classifyEntityStatus` in `MapSection.tsx` in lockstep тАФ the UI is the
mirror, not the source of truth.
agent ┬╖ 2026-05-04

### Job rows auto-expand when there's work to do
A job with any missing or partial component opens by default so the
user sees "what's not built yet" without an extra click. Fully-built
jobs collapse to keep the persona card scannable. This is a
per-render-mount choice (uses `useState` initial value, not a controlled
prop) so the user can manually collapse/expand and we don't fight them.
agent ┬╖ 2026-05-04

### `Project.tsx` keeps a `PLACEHOLDERS` map even though every tab is real
Cheap defense for when a future tab id is added to the union before
its section is wired up. The reviewer flagged this as dead code; we
keep it deliberately as the empty-state fallback.
agent ┬╖ 2026-05-04
